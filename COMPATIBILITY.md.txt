# Ma Pao's Homeschool Tutor — Cross-Platform Compatibility Policy

## 1. General Principles

- **Capability detection, never user-agent sniffing.**  
  Use `'speechSynthesis' in window`, `navigator.userAgentData`, `'SpeechRecognition' in window`, etc.  
  UA strings are unreliable and break over browser updates.

- **Graceful degradation.**  
  If a feature is unavailable, fail silently or show a plain-text fallback — never throw an unhandled error.

- **Pointer events first, mouse/touch as fallback.**  
  Use `onclick` or `addEventListener('pointerup', ...)`. Do not rely on `touchstart` alone.

## 2. Text-to-Speech (TTS) — `speechSynthesis` API

### Feature detection
```js
if (!('speechSynthesis' in window)) {
  // TTS not available — hide Read button, do nothing
  return;
}
```

### Known platform quirks and mitigations

| Platform / Browser | Quirk | Mitigation in code |
|---|---|---|
| **Edge (Chromium) desktop & mobile** | `speak()` silently fails: `onstart` never fires, no `onerror`, `speaking` stays `false` | 800 ms watchdog after `speak()` — if still not speaking, `cancel()` + retry once. Detected via `navigator.userAgentData.brands` (no UA string). |
| **Windows / Chrome / Edge** | `speechSynthesis` freezes mid-utterance (paused flag, or `speaking=true` but silent) | `resumeHack` interval: resumes if `paused`; restarts chunk if stuck >14 s with `isReading===false`. |
| **Windows / Chrome / Edge** | Second `cancel()` in same tick kills queued speech | Single `cancel()` in `readAloud()` / `startReadAloud()`, 400 ms delay before `speak()`. No second `cancel()` in `_doReadAloud`. |
| **iOS Safari** | `speak()` must be called synchronously inside a user gesture | `startReadAloud()` calls `doSpeak()` immediately (no `setTimeout`) when `hasPending` is true. |
| **All browsers** | `onstart` may fire late (especially Edge) | 1 s fallback timer in `speakChunk()` sets button state even if `onstart` hasn't fired. |
| **Firefox** | `onboundary` not implemented (word highlighting skipped) | `onboundary` handler is present but word index advances only on event — no highlight on Firefox; acceptable degradation. |
| **Android Chrome** | Voices list empty until `onvoiceschanged` fires | `ensureVoices()` waits for the event before calling `_doReadAloud`. |

### Audio codec fallbacks
This app uses the browser `speechSynthesis` API (no audio files are served). Codec fallbacks are therefore N/A. If a future version adds pre-recorded audio, serve both **MP3** and **OGG** and use `<audio>` `canPlayType()` to select:
```js
const audio = new Audio();
const src = audio.canPlayType('audio/ogg') ? 'file.ogg' : 'file.mp3';
```

### `play()` promise handling (for any future `HTMLAudioElement` use)
```js
const p = audio.play();
if (p !== undefined) {
  p.catch(err => {
    // Autoplay blocked — show a "Tap to play" prompt
    console.warn('Autoplay blocked:', err);
  });
}
```

## 5. Test Cases & Pass/Fail Criteria

### TC-01 · Read button — basic playback
- **Action:** Click/tap ▶ Read on a page with AI response text.
- **Pass:** Audio plays within 1.5 s; ▶ Read hides, ⏸ Pause appears.
- **Fail:** No audio after 3 s, or button stays as ▶ Read.

### TC-02 · Edge silent-fail watchdog
- **Action:** Click ▶ Read in Edge (desktop or mobile).
- **Pass:** Audio starts within 1.6 s (800 ms watchdog + 200 ms retry + buffer).
- **Fail:** No audio after 3 s.

### TC-03 · Pause / Resume
- **Action:** Click ⏸ Pause mid-speech; click ▶ Resume.
- **Pass:** Audio pauses and restarts at same position; buttons update correctly.
- **Fail:** Pause has no effect, or Resume restarts from beginning.

### TC-04 · Word highlighting
- **Action:** Click ▶ Read on Chrome or Edge.
- **Pass:** Words highlight in sync with speech; auto-scrolls to current word.
- **Fail:** No highlighting, or highlighting is static/wrong word.
- **Note:** Firefox — highlighting not expected (onboundary not supported); acceptable.

### TC-05 · Stop during playback
- **Action:** Navigate away or click ✗ Stop mid-speech.
- **Pass:** Audio stops immediately; buttons reset; no ghost speech continues.
- **Fail:** Audio continues or buttons stuck in playing state.

### TC-06 · Re-read (second tap without new response)
- **Action:** Tap ▶ Read, let it finish, tap ▶ Read again.
- **Pass:** Plays again from beginning without error.
- **Fail:** Silent fail, error in console.

### TC-07 · TTS on iOS (gesture requirement)
- **Action:** Open app cold in Safari iOS; tap ▶ Read without using mic first.
- **Pass:** Audio plays immediately on first tap.
- **Fail:** Silent fail; audio plays only after mic is used.

### TC-08 · No `speechSynthesis` — graceful degradation
- **Action:** Simulate `window.speechSynthesis = undefined` via devtools.
- **Pass:** Read button either hidden or click does nothing; no JS error.
- **Fail:** Uncaught exception thrown.

### TC-09 · resumeHack — stuck speech recovery
- **Action:** On Windows/Chrome, let a long response play; observe console for hack firing.
- **Pass:** Speech continues without going silent; hack fires < once per session.
- **Fail:** Speech goes permanently silent mid-text with hack not recovering it.

### TC-10 · Race condition — rapid button clicks
- **Action:** Click ✓ Correct, ✗ Wrong, Hint, or Send buttons multiple times rapidly.
- **Pass:** Only one AI evaluation fires; buttons re-enable after response.
- **Fail:** Multiple API calls fire, or buttons stay disabled permanently.

### TC-11 · Last Lesson saves after session ends
- **Action:** Open any subject for any student, begin a session, answer at least one question, let the timer complete (or wait for `endSessionByTimer()`).
- **Pass:** Open Teacher Dashboard → subject row shows updated "Last Lesson" text (not "Not started" or "—"). Value matches what the AI reported as the lesson topic.
- **Fail:** Last Lesson still shows "Not started" or "—" after session ends.
- **Code at risk:** `endSessionByTimer()`, `saveSession()`, `sess.lastLesson` assignment.

### TC-12 · Session count increments exactly once per session
- **Action:** Open a subject, begin session, complete it (timer ends or all questions answered). Check session count in dashboard.
- **Pass:** Count increases by exactly 1. Open the same subject again and complete another session — count increases by 1 again.
- **Fail:** Count increases by 2 (double-increment), stays at 0, or resets to 0 on reload.
- **Code at risk:** `sessionCountedThisOpen` guard in `endSessionByTimer()` and `markResponse()`. Must be `false` at session start and set to `true` on first increment.

### TC-13 · Progress persists after page reload (localStorage)
- **Action:** Complete a session so Last Lesson and session count are updated. Hard-reload the page (Ctrl+Shift+R). Open Teacher Dashboard.
- **Pass:** Last Lesson and session count are identical to before the reload.
- **Fail:** Values reset to "Not started" / 0 after reload.
- **Code at risk:** `saveSession()` writing to `localStorage`; `getSession()` reading from `localStorage` on render.

### TC-14 · Cross-device sync — Supabase round-trip
- **Action:** Complete a session on Device A (e.g. Windows Chrome). Open the app fresh on Device B (e.g. iPhone Safari or Linux Chrome). Open Teacher Dashboard on Device B.
- **Pass:** Last Lesson and session count from Device A appear on Device B within a few seconds of opening.
- **Fail:** Device B shows "Not started" / 0 — local data from Device A never reached Supabase, or Device B never loaded from Supabase.
- **Code at risk:** `syncToSupabase()` called in `saveSession()` and `toggleDone()`; `loadSessionsFromSupabase()` called on startup and in `selectStudent()` (must be awaited before `renderSubjects()`).

### TC-15 · Done Today resets at midnight
- **Action:** Mark a subject as done (✓). Note the date. The next calendar day, reload the app and open Teacher Dashboard.
- **Pass:** The subject shows "—" (not done) under Today column on the new day.
- **Fail:** Subject still shows "✔ Done" the next day.
- **Code at risk:** `getDoneKey()` uses date string `YYYY-M-D` as part of the localStorage key — a new day produces a new key, so old "done" state is invisible. Verify date string format has not changed.

### TC-16 · Dashboard manual edit saves and syncs
- **Action:** In Teacher Dashboard, click ✏️ on any subject row. Change Last Lesson to "Lesson 7" and Sessions to 3. Click ✔ Save.
- **Pass:** Row immediately shows "Lesson 7" and 3 sessions. Reload the page — values persist. On a second device, reload dashboard — values match (Supabase synced).
- **Fail:** Row reverts after save, values lost on reload, or second device shows old values.
- **Code at risk:** `saveDashEdit()` writing to localStorage and calling `syncToSupabase()`; `renderDashRow()` re-rendering the row after save.

### TC-17 · Cloud wins over local when cloud session count is higher
- **Action:** Manually set a student's session count to 5 in Supabase Table Editor. Clear localStorage (`localStorage.clear()` in browser console). Reload the app.
- **Pass:** Teacher Dashboard shows 5 sessions for that subject — cloud data was loaded into localStorage on startup.
- **Fail:** Dashboard shows 0 — `loadSessionsFromSupabase()` did not seed localStorage, or the cloud-wins logic was bypassed.
- **Code at risk:** Merge logic in `loadSessionsFromSupabase()`: `if(cloudCount > localCount)` overwrites local. Verify this condition has not been inverted or removed.

### TC-18 · FC inherits FA subject changes
- **Action:** Add a new flag or subject to F.A.'s subject list in code (e.g. a new `isAesop: true` flag). Deploy. Open F.C.'s subject list in the app.
- **Pass:** F.C. has the identical subject with the identical flag — no separate code change was needed.
- **Fail:** F.C. is missing the subject or flag — inheritance via object spread was broken.
- **Code at risk:** F.C. subjects defined as `...FA_SUBJECT_OBJECT_SPREAD` at runtime. Any change that hardcodes F.C. subjects separately breaks this.

## 7. Supabase — Known Issues & Troubleshooting

### 7.1 Free-tier auto-pause
Supabase free-tier projects pause automatically after **7 days of inactivity** (no database requests).

**Symptom:** Dashboard shows red banner — `Cloud error: HTTP 401` or `HTTP 500`.

**Prevention:** `supabaseKeepAlive()` in `index.html` sends a HEAD ping to Supabase every 23 hours.
- This function and its startup call **must never be removed** from `index.html`.
- Verify its presence after every edit with: `grep -n "supabaseKeepAlive" index.html`

**Recovery if paused:**
1. Go to [supabase.com](https://supabase.com) → sign in → open `mapao-progress` project
2. If banner says **"Project paused"** → click **Restore** and wait ~2 minutes
3. Reload the app — cloud sync should resume automatically

---

### 7.2 API key format (important — June 2026 migration)
Supabase migrated to a new `sb_publishable_...` key format. **This format does NOT work** with the direct REST `apikey` / `Authorization: Bearer` header pattern used in this app.

**Always use the legacy `eyJ...` JWT key:**
- In Supabase dashboard → Settings → API Keys → click **"Legacy anon, service_role API keys"** tab
- Copy the **anon** key (starts with `eyJhbGciOiJIUzI1NiIs...`)
- This is set as `SUPA_KEY` in `index.html`

**Current key location in code:** `const SUPA_KEY = 'eyJ...'` (line ~776)

---

### 7.3 HTTP 401 — authentication rejected
**Causes (in order of likelihood):**
1. Wrong key format — using `sb_publishable_` instead of legacy `eyJ...` JWT (see 7.2)
2. Project is paused (see 7.1)
3. Key was rotated in Supabase dashboard — fetch new legacy key and update `SUPA_KEY`

---

### 7.4 HTTP 403 — key valid but access denied (RLS)
Row Level Security is blocking the request even though the key is correct.

**Fix:**
1. Supabase dashboard → Authentication → Policies → find `progress` table
2. Click **"Disable RLS"** (safe for this private app — no user auth is needed)

---

### 7.5 "API DISABLED" — table not exposed to REST
The `progress` table exists but is not reachable via the REST API.

**Symptom:** Policies page shows `API DISABLED` badge next to the `progress` table.

**Fix:**
1. Supabase dashboard → Integrations → Data API → **Settings** tab
2. Click **"Exposed tables"** dropdown → toggle `progress` **on**
3. Click **Save** → confirm "Successfully saved settings"

**Note:** This can happen when a new project is created or after certain Supabase dashboard migrations. Always verify after any Supabase project reset.

## 9. Commit Reference


**Branch:** `fix/edge-tts-silent-fail`  
**Files changed:** `index.html` (2 locations)  
**Commit message:**
```
fix(tts): add Edge silent-fail watchdog + capability-detect IS_EDGE_CHROMIUM

Edge (Chromium) silently drops speechSynthesis.speak() with no error.
Detected via navigator.userAgentData.brands (no UA sniffing).
800ms watchdog after speak(): if still not speaking, cancel + retry once.
Also adds IS_EDGE_CHROMIUM constant using UA Client Hints API.

Closes: Edge Read button not working (desktop + mobile)
```

**PR description:**
> **Problem:** On Microsoft Edge (desktop and mobile), clicking ▶ Read produces no audio and no error. `speak()` is called but `onstart` never fires and `speechSynthesis.speaking` stays `false`.
>
> **Fix:** After `speak()`, a 800 ms watchdog checks if synthesis actually started. If not, it cancels and retries the chunk once. Applies only when `IS_EDGE_CHROMIUM` is true (detected via `navigator.userAgentData.brands`, not UA string).
>
> **No behaviour changes** on Chrome, Firefox, or Safari. All five async buttons remain managed together per the existing race-condition fix.
